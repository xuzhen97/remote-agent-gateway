# 客户端列表页版本号展示设计

**日期**: 2026-06-14  
**范围**: `apps/web` 客户端列表页展示调整

## 背景

当前 Web 控制台的“客户端”列表页未展示客户端版本号，但后端 `/api/clients` 已返回 `version` 字段，前端 `ClientSummary` 类型也已定义 `version?: string`。因此问题不在数据链路，而在列表页 UI 未消费该字段。

用户确认的目标界面为 Web 控制台“客户端”列表页，截图显示当前列包含：ID、名称、在线、HTTP 就绪、HTTP 地址、能力、操作。

## 目标

在“客户端”列表页中新增独立“版本”列，便于批量查看各客户端当前版本。

## 非目标

本次不修改以下内容：

- 服务端 `/api/clients` 返回结构
- 客户端注册与版本上报逻辑
- 客户端详情页布局
- 仪表盘、更新管理页的版本展示逻辑
- 版本筛选、排序、搜索等增强功能

## 方案比较

### 方案 A：新增独立“版本”列

在“名称”后新增“版本”列，直接显示客户端版本号。

**优点**
- 语义清晰，版本是独立资产信息
- 便于批量扫描和横向对比
- 与仪表盘中已有版本展示认知一致

**缺点**
- 表格宽度略有增加

### 方案 B：合并到“名称”列

名称下方附带显示版本号。

**优点**
- 不增加列宽

**缺点**
- 扫描效率较低
- 不利于后续扩展排序/筛选

### 方案 C：放入“能力”区域作为 Tag

在能力标签区追加版本信息。

**优点**
- 紧凑

**缺点**
- 语义不准确，版本不属于能力
- 可读性差

## 最终决策

采用 **方案 A：新增独立“版本”列**。

## 详细设计

### 展示位置

在 `apps/web/src/pages/ClientsPage.tsx` 的表格列配置中，于“名称”后插入“版本”列。

建议列顺序：
1. ID
2. 名称
3. 版本
4. 在线
5. HTTP 就绪
6. HTTP 地址
7. 能力
8. 操作

### 展示样式

- 有版本号时：使用紧凑、易识别的代码风格文本展示，如 `v0.1.0`
- 无版本号时：显示 `-`

为与当前控制台其他页面保持一致，优先复用 Ant Design 的 `Typography.Text`，并使用 `code` 样式。

### 数据来源

直接使用现有 `ClientSummary.version` 字段，无需新增接口字段。

### 容错处理

- `version` 为 `undefined`、空字符串或不可用时，前端统一显示 `-`
- 不在前端做额外版本格式化逻辑，除前缀展示 `v` 外不引入语义转换

## 测试设计

在 `apps/web/src/pages` 下新增或补充 `ClientsPage` 相关测试，至少覆盖：

1. 当接口返回 `version: '0.1.0'` 时，页面渲染 `v0.1.0`
2. 当接口未返回版本时，页面渲染 `-`

测试目标是验证列表页渲染行为，不测试后端接口实现。

## 影响范围

### 修改文件

- `apps/web/src/pages/ClientsPage.tsx`
- `apps/web/src/pages/ClientsPage.test.tsx`（新增）

### 不需要修改的文件

- `apps/web/src/api/clients.ts`（已有 `version` 定义）
- `apps/server/src/modules/clients/*`（已返回版本字段）
- `apps/client/*`（无需变更）

实现后应至少运行：

- `pnpm --filter @rag/web test`
- `pnpm --filter @rag/web typecheck`

如进行整仓收尾验证，再运行：

- `pnpm typecheck`
- `pnpm test`

## 风险与回退

### 风险

- 表格列宽增加后，窄屏下可能更拥挤
- 若版本值为空字符串，需保证渲染降级为 `-`

### 回退

若展示效果不理想，可仅回退 `ClientsPage.tsx` 中新增列及对应测试，不影响数据链路。
