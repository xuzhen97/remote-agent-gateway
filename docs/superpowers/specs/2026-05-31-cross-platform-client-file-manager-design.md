# 跨平台客户端文件管理器设计

日期：2026-05-31

## 背景

当前远程文件管理只覆盖客户端 `workspaceDir`，并且前端入口偏轻量，无法满足“像真正文件管理器一样浏览和管理节点文件系统”的需求。

目标是把客户端文件管理扩展为一个受权限约束的、跨平台一致的文件管理器：

- Windows：先列出允许访问的盘符或根目录，再向下浏览目录与文件。
- Linux：先列出允许访问的根目录，例如 `/`、`/var/log`、`/workspace`，再向下浏览。
- 权限原则：不做提权；仅允许操作当前客户端进程有权限访问的路径。
- 架构原则不变：WebSocket 仍是控制面，HTTP/FRP 仍是文件数据面。

## 设计结论

采用 `allowedRoots` 多根目录模式，而不是完全放开系统根目录，也不是只在 `workspaceDir` 与“系统模式”之间二选一。

推荐原因：

- 同时支持 Windows 盘符与 Linux 根目录风格。
- 安全边界清晰，避免“全盘可见”的默认风险。
- 便于后续做生产环境配置与审计。
- 前端可以用统一的“先选根，再输入相对路径”的模型，避免平台差异直接暴露到 UI。

## 范围

本设计覆盖：

- 客户端多根目录配置模型
- 服务端与客户端文件 API 的路径语义调整
- 前端客户端文件管理器交互改造
- 权限语义、错误码、测试策略

本设计不覆盖：

- 提权运行
- 全盘索引与搜索
- 文件预览器、diff 预览、批量选择
- 高级 ACL 展示
- 断点续传与大文件专用协议

## 方案对比

### 方案 A：完全放开系统根目录

优点：

- 最接近传统文件管理器。
- 不需要维护额外根列表。

缺点：

- 风险最高。
- 不同平台上的系统目录差异、权限错误和误操作成本都更高。
- 对生产环境不友好。

结论：不采用。

### 方案 B：`workspaceDir` 与系统模式二选一

优点：

- 兼容现有逻辑，实现成本较低。

缺点：

- 模式切换心智负担大。
- “系统模式”一旦打开仍然边界过宽。
- 前端和后端都需要额外模式分支。

结论：不采用。

### 方案 C：`allowedRoots` 多根目录模式

优点：

- 边界清晰，安全可控。
- 同时支持 Windows 和 Linux。
- 前端可用统一交互模型。
- 便于按节点做不同可见范围配置。

缺点：

- 需要引入根列表接口与 `rootId` 语义。
- 客户端路径解析逻辑比单根目录稍复杂。

结论：采用。

## 配置模型

客户端配置增加 `allowedRoots`，用于声明当前节点允许浏览的根目录集合。

### Windows 示例

```json
{
  "allowedRoots": [
    "C:\\",
    "D:\\data",
    "./workspace"
  ]
}
```

### Linux 示例

```json
{
  "allowedRoots": [
    "/",
    "/var/log",
    "./workspace"
  ]
}
```

### 默认行为

如果未配置 `allowedRoots`：

- 自动回退到当前 `workspaceDir` 作为唯一允许根。
- 这样现有节点无需立刻修改配置也能继续工作。

### 路径解析规则

- 相对根项，如 `./workspace`，以 `config.json` 或客户端运行配置目录为基准解析。
- Windows 下允许 `C:\`、`D:\projects` 这类绝对路径。
- Linux 下允许 `/`、`/home/user` 这类绝对路径。
- 所有根在加载时标准化为绝对路径，并生成稳定的 `rootId`。

## 数据模型

新增“根项”概念：

```ts
interface ClientFileRoot {
  id: string;
  label: string;
  path: string;
}
```

其中：

- `id`：前端与接口交互使用的稳定标识。
- `label`：面向用户展示，例如 `C:\`、`D:\data`、`workspace`、`/var/log`。
- `path`：客户端本地真实绝对路径，仅在服务端/客户端内部使用。

## API 设计

### 新增根列表接口

```http
GET /api/clients/:clientId/files/roots
```

返回当前客户端可浏览的根列表。

示例：

```json
[
  { "id": "root-0", "label": "C:\\", "path": "C:\\" },
  { "id": "root-1", "label": "D:\\data", "path": "D:\\data" },
  { "id": "root-2", "label": "workspace", "path": "D:\\remote-agent-gateway\\apps\\client\\workspace" }
]
```

### 现有文件接口调整为 `rootId + path`

所有客户端文件接口从“单独的 path”扩展为“根 + 相对路径”。

示例：

```http
GET /api/clients/:clientId/files?rootId=root-0&path=Windows/System32
GET /api/clients/:clientId/files/read?rootId=root-1&path=nginx/access.log
PUT /api/clients/:clientId/files/write?rootId=root-2&path=reports/a.txt
POST /api/clients/:clientId/files/upload?rootId=root-2&path=reports&filename=b.txt
```

JSON body 类接口同样加入 `rootId`：

```json
{
  "rootId": "root-2",
  "path": "reports/2026",
  "recursive": true
}
```

### 保持控制面/数据面架构不变

- WebSocket：继续负责启动客户端 file session、FRP 映射与任务协调。
- HTTP/FRP：继续负责文件内容传输与目录读写操作。

本次不改变整体控制面/数据面分层。

## 客户端实现模型

客户端本地 HTTP 文件服务不直接接受任意绝对路径，而是接收：

- `rootId`
- `path`

处理流程：

1. 根据 `rootId` 找到对应的真实根路径。
2. 将 `path` 解析为该根下的绝对路径。
3. 校验解析结果仍落在该根范围内。
4. 调用本地文件系统执行读写删操作。
5. 如果权限不足，返回明确的权限错误，而不是泛化为内部错误。

### 关键原则

- 前端不直接提交绝对路径给客户端。
- 客户端本地服务永远是最终路径边界的判定者。
- 路径校验与权限校验必须分开处理。

## 前端交互设计

客户端文件管理页改造成真正的双栏文件管理器。

### 总体布局

- 左栏：根入口列表
- 右栏：当前根下的目录与文件视图

### 左栏：根入口

展示当前节点的 `allowedRoots`：

- Windows：`C:\`、`D:\data`、`workspace`
- Linux：`/`、`/var/log`、`workspace`

用户点击根入口后切换当前浏览根。

### 右栏：文件视图

顶部包含：

- 当前路径栏
- 打开路径
- 上级
- 刷新
- 新建目录
- 新建文件/写入
- 上传到当前路径

主表格包含：

- 名称
- 类型
- 大小
- 修改时间
- 操作

### 交互规则

1. 先选节点，再选根。
2. 切换节点时路径重置为默认根的 `.`。
3. 切换根时路径重置为 `.`。
4. 目录通过“进入”按钮向下浏览。
5. 通过“上级”按钮返回父目录。
6. 上传的语义固定为“上传到当前节点当前路径”。

### 交互实现要求

- 不再使用浏览器原生 `prompt()` / `confirm()`。
- 使用页面内 modal 或表单。
- 原因：
  - 跨平台浏览器行为更一致。
  - 自动化测试更稳定。
  - 更容易扩展错误提示与校验逻辑。

### 服务端文件仓库定位

`文件` 页继续保留：

- `服务端文件仓库`
- `客户端文件`

其中：

- 服务端文件仓库：适合中转、共享、归档
- 客户端文件：适合作为节点文件系统直接管理入口

客户端文件应成为主流程入口。

## 权限语义与错误处理

### 错误分类

#### 1. 越界错误

场景：

- `rootId` 不存在
- `path` 解析后跑出对应根目录之外

返回：

- `400`

文案：

- `Path outside allowed root`

#### 2. 权限错误

场景：

- 路径在允许根内，但当前进程无权限访问或写入

返回：

- `403`

文案：

- `Permission denied`

#### 3. 不存在

场景：

- 文件或目录不存在

返回：

- `404`

#### 4. 冲突

场景：

- 目录已存在
- move/copy 目标存在且不允许覆盖

返回：

- `409`

#### 5. 内部异常

场景：

- 真正的服务端/客户端内部错误

返回：

- `500`

### 前端展示原则

- 目录读取失败时，在文件区域保留可见错误提示，而不是只弹 toast。
- 写入/删除/上传失败时使用 toast，但必须显示明确原因。
- 对无权限根或无权限目录，前端要能区分“目录不存在”和“无权限访问”。

## 跨平台兼容策略

### 可执行文件路径

客户端 `frpcPath`：

- 支持相对路径配置，如 `./bin/frpc`
- 自动从配置目录向上搜索
- Windows 自动补 `.exe`
- 如果配置未写绝对路径，也尽量能在 monorepo 或打包目录中找到实际二进制

服务端 `FRPS_BIN_PATH`：

- 同样支持相对路径
- 自动向上搜索 `bin/frps` 或 `bin/frps.exe`
- 避免因 cwd 不同导致 builtin FRP 启动失败

### 路径展示

前端统一展示“根标签 + 相对路径”，不要求用户输入系统绝对路径。

### 浏览器交互

避免原生对话框，统一改为页面内 modal。

## 测试策略

### 1. 共享/协议层

增加 `rootId + path` schema 测试，覆盖：

- Windows 风格路径
- Linux 风格路径
- 越界路径

### 2. 客户端配置与路径解析

- `frpcPath` 自动向上搜索
- Windows 自动补 `.exe`
- 相对路径基于配置目录解析

### 3. 客户端本地文件服务

覆盖：

- 允许根内读写成功
- 越界拒绝
- 权限不足返回 `403`
- 不存在返回 `404`

### 4. 服务端路由/代理测试

覆盖：

- `roots` 接口返回正确根列表
- `rootId + path` 请求正确转发
- 浏览器上传二进制 body 可正确透传
- 错误码透传正确

### 5. E2E

覆盖：

- 根列表展示
- 进入目录
- 上传文件
- 写文件
- 删除文件
- 权限拒绝

重点证明：

- WebSocket 仍是控制面
- HTTP/FRP 仍是文件数据面

## 实现顺序建议

1. 后端/客户端 `allowedRoots` 配置模型
2. roots API
3. 客户端本地路径解析改为 `rootId + path`
4. 权限与错误码语义
5. 前端根列表 + 文件管理器双栏视图
6. UI modal 改造
7. E2E 与跨平台验证

## 风险与边界

### 风险 1：根目录过宽

如果 `allowedRoots` 配成 `C:\` 或 `/`，虽然逻辑受限于当前进程权限，但依然会带来更大的操作面。

应对：

- 默认仍回退到 `workspaceDir`
- 由使用者显式决定是否打开更宽根目录

### 风险 2：旧 FRP file-service 映射残留

文件会话如果不清理旧映射，可能出现新返回端口不通的情况。

应对：

- 启动新的 file session 前先清理旧的 file-service 映射
- 等待 remove/create 任务完成后再返回新 session

### 风险 3：跨平台路径差异

Windows 盘符、Linux 根路径、分隔符、`.exe` 差异都可能导致运行环境不一致。

应对：

- 所有路径解析集中在客户端/服务端统一 helper
- 所有 UI 只操作 `rootId + path`
- 可执行文件路径自动探测

## 非目标

本轮不做：

- 全局搜索
- 预览器
- 文件 diff
- 批量操作
- ACL 明细展示
- 提权访问

## 验收标准

当满足以下条件时认为设计目标达成：

- 前端可先选节点，再选允许根目录，再浏览目录。
- Windows 可从配置允许的盘符根开始管理。
- Linux 可从配置允许的根目录开始管理。
- 越界访问被拒绝。
- 无权限访问返回明确错误。
- 上传/写入/删除只在当前进程权限允许时成功。
- 浏览器交互不依赖原生 `prompt()` / `confirm()`。
- FRP/HTTP 文件数据面架构保持不变。
